# Make.com — Install 4 Cockpit Blueprints (01, 02, 03, 05)

These blueprints pair with the GH Actions weekly digest (already running). Build order: **04 first (already shipped) → 01 → 05 → 02 → 03**.

## What you need before importing

| Blueprint | Required tokens / connections |
|---|---|
| **01 Deploy Watcher** | `VERCEL_TOKEN` (https://vercel.com/account/tokens), Supabase service key |
| **02 Uptime Watcher** | `GITHUB_PAT` (repo:write scope), Supabase service key, an uptime monitor (UptimeRobot free tier or Better Stack) |
| **03 Email Intake** | Gmail OAuth in Make, Anthropic API key, Supabase service key, dev intake email address provisioned |
| **05 Incident Logger** | `GITHUB_PAT` (repo:write scope), Supabase service key |

## Generic install steps (apply to each)

1. **Make.com → Scenarios → Create new scenario → ⋯ menu → Import Blueprint**
2. Upload the `.blueprint.json` file.
3. Open each HTTP module that has `REPLACE_WITH_*` tokens. Replace inline OR (better) move into Make → **Custom Variables** so they're not in the scenario JSON.
4. For the webhook module: click it → **Add hook** → name it as labelled (e.g. `vercel-deploy-event`) → **copy the URL** (you'll paste it back into the source system).
5. Click **Run once** to test. If the scenario expects a webhook payload, send a test from the source.
6. Toggle the scenario **ON**.

## Blueprint-specific setup

### 01 Deploy Watcher
After import:
1. Webhook URL → paste into Vercel: **Project Settings → Git → Deploy Hooks → Add Hook** (subscribe to `deployment.error`, `deployment.canceled`, `deployment.succeeded`).
2. Test path: push a deliberately broken commit to a branch with Vercel preview enabled. Watch scenario fire on `deployment.error`. Confirm `cockpit_incidents` row created and the previous prod deployment got promoted (auto-rollback).

### 02 Uptime Watcher
**Pre-requisite**: pick an uptime monitor first.
- **UptimeRobot free** (recommended, 5-min check interval, free): https://uptimerobot.com/signUp — add monitor for `https://namkhan-bi.vercel.app` → **Alert Contacts → Webhook**.
- **Better Stack** (also free tier): https://betterstack.com/

After import:
1. Webhook URL → paste into UptimeRobot/Better Stack alert contact.
2. The monitor must POST a JSON body with at minimum `{ "alertType": "down|up", "url": "...", "monitor": "namkhan-bi" }`. Map fields in the alert contact accordingly.
3. Test path: pause your monitor or use the monitor's "send test alert" feature.

### 03 Email Intake
**Pre-requisite**: dev intake email provisioned. Options:
- Gmail alias (e.g. `pbsbase+dev@gmail.com`) — easiest, no DNS work.
- Custom domain alias (e.g. `dev@thedonnaportals.com`) — better long-term, needs Google Workspace or Mailgun forwarding.

After import:
1. Replace `REPLACE_WITH_DEV_INTAKE_EMAIL` in module 1's `query` field with the actual email address.
2. Connect Gmail (module 1 + module 4) — Make does OAuth in-UI, takes 30s.
3. Test path: send an email to the intake address. Wait up to 5 min (the trigger interval). Confirm new `cockpit_tickets` row + ack reply received.

### 05 Incident Logger
After import:
1. Webhook URL → use as the destination for any source that should log incidents:
   - **Vercel monitoring rules** (Phase 5) — paste as alert webhook.
   - **Supabase Database Webhooks** — for advisor changes / RLS violations.
   - **GitHub Actions** dependency check — already configured to call this URL via `MAKE_AUDIT_WEBHOOK` if you want (or use repository_dispatch).
2. The webhook must POST `{ "source": "...", "symptom": "...", ...rest }`. The scenario maps source → severity.
3. Test path: `curl -X POST <webhook-url> -H "Content-Type: application/json" -d '{"source":"dependency_high_vuln","symptom":"test S3"}'`. Confirm `cockpit_incidents` row + GitHub issue (only fires for severity ≤ 2).

## Caveats

- Make.com's blueprint JSON format is not fully documented publicly. Import sometimes succeeds with quirks. If a module imports broken: rebuild that module from scratch using the JSON as reference.
- Move all `REPLACE_WITH_*` tokens into Make Custom Variables (Org → Custom Variables) so they're not in any scenario re-export.
- Webhook URLs are **secret** — anyone with the URL can fire your scenario. Don't paste them into git or chat.
- For **03 Email Intake**: the "Claude Code Web" trigger originally specified is not yet built. This blueprint instead does the triage inline and creates the ticket directly. The Dev Arm picking it up to write code is still TODO (handled by the `/cockpit` chat triage layer for now).
