4. Pricing Recommender — OTB-aware floor logic

Decision flow for every BAR proposal

1
Read OTB for the target date
From Pace Agent output: rooms OTB, lowest refundable rate, weighted avg rate, pace vs STLY, days to arrival
2
Pace check (BLOCKER)
If OTB_pace_vs_STLY > +5% AND days_to_arrival < 30 → BLOCK rate drops, propose RAISE only
3
Hard floor (cannot breach)
floor = MAX(static_floor_per_room_type, lowest_refundable_OTB_rate) — proposal below floor → REJECT silently
4
Soft floor (warn, requires human override)
soft_floor = weighted_avg_OTB_rate × 0.92 — proposal below soft_floor → flag red, human must explicitly approve
5
Demand signal weighting
Comp set delta + Google Trends direction + events. Trends used as tiebreaker only — never as standalone trigger
6
Apply guardrails (next section)
Validate against rate change rules table. If any rule fails → mark as "deferred to next cycle"
7
Write to recommendation sheet
Sheet column: date, room type, current BAR, proposed BAR, delta %, rationale (≤80 chars), confidence, signals used

Pseudocode

# 13_OTB_FLOOR_LOGIC.md — pseudocode for Pricing Recommender def propose_bar(date, room_type): otb = pace_agent.read_otb(date, room_type) current_bar = cloudbeds.get_current_bar(date, room_type) # Step 2: Pace blocker if otb.pace_vs_stly_pct > 5.0 and otb.days_to_arrival < 30: direction = "raise_only" else: direction = "both" # Step 3: Hard floor static_floor = pricing_rules.floor[room_type] hard_floor = max(static_floor, otb.lowest_refundable_rate) # Step 4: Soft floor soft_floor = otb.weighted_avg_rate * 0.92 # Compute target rate from demand signals target = compute_target_bar(current_bar, demand_signals, direction) # Apply floors if target < hard_floor: return reject("below hard floor") if target < soft_floor: flag = "requires_human_override" # Apply magnitude guardrail max_change = current_bar * 0.15 target = clamp(target, current_bar - max_change, current_bar + max_change) return recommendation(date, room_type, target, flag, signals)

10. Open risks (still real, even with no writes)

Bad data in Reporting Agent → wrong decisions even without auto-pricing
Recommend-only doesn't protect you if the human trusts wrong numbers. Mitigation: Team Lead validates every KPI cell against 05_KPI_DEFINITIONS.md, refuses any cell it can't trace. Add a daily audit row: "X rooms in PMS folio = Y rooms in Reporting Agent output." Mismatch > 0.1% triggers red alert before flash sends.
Comp set decay — manual entry skipped or sloppy
By week 4, the Friday shop becomes "5 minutes before going home." Bad data feeds bad recommendations. Mitigation: agent refuses to issue recommendations if comp set sheet is >10 days stale, alerts human RM.
Human override fatigue
If the human approves 95% of recommendations, autonomy is happening de-facto without the safety. Mitigation: track accept/override rate. If >90% accept after 30 days, force a "challenge week" — human must override at least 3 recommendations with documented rationale.
OTB-aware logic depends on Pace Agent accuracy
If Pace Agent miscomputes OTB (e.g., misses last-minute bookings), Pricing Recommender makes wrong decisions on correct logic. Garbage in, garbage out. Mitigation: Pace Agent runs sanity check — total OTB rev = sum of folio rooms revenue. Mismatch → flag and skip pricing cycle that day.
Google Trends signal noise
Trends index swings ±20% week-to-week even with stable demand. Treating it as direction signal still risks false positives. Mitigation: 4-week rolling average only · used only as tiebreaker between two close decisions, never as primary trigger.
Promo Analyzer attribution problem
"Did this promo cause incremental bookings, or would they have come anyway?" is genuinely hard. Pre/post comparison ignores seasonality. Mitigation: agent reports outputs as "indicators" not "causal" — use 90-day pre/post + same-period LY as control. Never claim causation in the report wording.
